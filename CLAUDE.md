## What This Is

HotSwitch-HS is a Hammerspoon Lua module for macOS window switching using 2-stroke hotkeys. It has no build system, no package manager, and no test framework ‚Äî it's pure Lua running inside Hammerspoon.

## Development Workflow

1. Edit `.lua` files.
2. Reload Hammerspoon config to test: click the Hammerspoon menubar icon ‚Üí `Reload Config`, or use a keybind like `hs.hotkey.bind({"command", "option", "control"}, "r", hs.reload)`.
3. Check errors in Hammerspoon Console (menubar icon ‚Üí `Console...`).

The module is loaded from `~/.hammerspoon/init.lua` via `require("hotswitch-hs/hotswitch-hs")`.

## Architecture

MVC pattern. Entry point `hotswitch-hs.lua` exposes the public API and wires up a `MainController`.

**Controllers**
- `MainController` ‚Äî orchestrates open/close lifecycle: enables hotkeys, activates the Hammerspoon canvas window, refreshes windows, shows the panel, and watches for app deactivation.
- `HotkeyController` ‚Äî owns all `hs.hotkey` objects. Hotkeys are created once at startup but only **enabled** while the panel is open (for performance). Handles key actions: navigation (Tab/arrows), registration mode toggle (Space), focus (Return/char keys), delete (Delete), cancel (Escape).

**Models**
- `WindowModel` ‚Äî wraps `hs.window.filter` to get ordered windows. Subscribing to `windowVisible` events is required for performance ‚Äî without it, `getWindows()` is slow. Caches the ordered window list. Contains complex `focusWindow()` logic to work around a [Hammerspoon bug](https://github.com/Hammerspoon/hammerspoon/issues/370) where `window:focus()` fails for multi-window apps across screens (uses `hs.timer.doAfter(0.01)` double-focus + `raise()` workarounds).
- `SettingModel` ‚Äî persists registered key assignments via `hs.settings` under key `"hotswitch-hs"`. Data is keyed by app **bundleID** (not app name). Format: `{ { app = "com.apple.Safari", keys = {"s", "f"} }, ... }`.
- `KeyStatusModel` ‚Äî builds the runtime key assignment table by merging `SettingModel` data with auto-generated keys from `KeyConstants.DEFAULT_AUTO_GENERATED_KEYS`. Separates `registeredKeyStatuses` from `registeredAndAutoGeneratedKeyStatuses`.
- `AppWatchModel` ‚Äî watches for application deactivation events to auto-close the panel.

**Views**
- `PanelLayoutView` ‚Äî composes `BaseCanvasView` (the full window list canvas) and `SelectedRowCanvasView` (the highlight row). Tracks `isOpen` state.
- `BaseCanvasView`, `SelectedRowCanvasView` ‚Äî render using `hs.canvas`.
- `ToastView` ‚Äî transient notification overlay.

**Common**
- `KeyConstants` ‚Äî defines `BASIC_KEYS`, `SHIFTABLE_KEYS`, `DEFAULT_AUTO_GENERATED_KEYS`, and `ADDITIONAL_LAYOUT.JAPANESE`.
- `FrameCulculator` ‚Äî computes panel positioning; `setShowingOnMainScreen(false)` makes panel follow primary screen.
- `Debugger`, `TimeChecker`, `Updater` ‚Äî logging, perf timing, and `git pull` auto-update.

## Key Behaviors to Be Aware Of

- **Finder windows** get special handling: only the most recently focused Finder window is shown, and focusing Finder uses `hs.application.launchOrFocusByBundleID` instead of `window:focus()`. **Do not attempt to show all Finder windows** ‚Äî see the Finder limitations section below for why this is not solvable with the current approach.
- **Lua garbage collection**: the README warns to pay attention to this. Hammerspoon objects (hotkeys, watchers, canvas) must be kept alive by storing them in variables with sufficient scope.
- **Log level defaults to `"nothing"`** to avoid a macOS Ventura performance bug where printing many logs in Hammerspoon console causes slowness.
- Settings are stored persistently by Hammerspoon's `hs.settings` ‚Äî use `hotswitchHs.clearSettings()` to reset key registrations during development.

## Release Workflow

When making a releasable change:

1. Add a changelog entry to `README.md` under `# ChangeLogs` (newest entry at the top).
2. Commit `README.md` together with the changed source files in a single commit.
3. Tag the commit with the new version: `git tag vX.Y.Z`
4. Version bumping: increment the **patch** version by default (`v2.5.4` ‚Üí `v2.5.5`) unless otherwise specified.

## Finder Window Limitations

Showing all real Finder windows in the panel was attempted (commit `a2e893a`, then reverted) and could not be made to work reliably. The constraints are fundamental to how Hammerspoon and Finder interact:

**Stale subscription cache (phantom windows):**
`hs.window.filter` uses a `windowVisible` event subscription for performance ‚Äî without it, `getWindows()` is prohibitively slow. When the user merges Finder windows into a tab group, Finder does not fire the proper `AXUIElement` hidden/destroyed accessibility notification. The filter's internal cache therefore retains the now-phantom window objects from before the merge. These phantoms are indistinguishable from real windows via the subscription cache. A fresh accessibility query (`hs.window.filter:getWindows()` without cache, or `application:visibleWindows()`) can identify real windows, but the IDs returned by these fresh queries may not match the IDs stored in the subscription cache, causing real windows to be incorrectly dropped.

**`window:focus()` resets Finder to first tab:**
`window:focus()` internally calls `becomeMain()` (sets `AXMain = true`), which causes Finder to jump to the first tab in the tab group instead of preserving the currently active tab. Using `raise()` + `application:activate(true)` avoids the `AXMain` side-effect but does not resolve the phantom-entry problem above.

**Current approach:** `removeUnusableFinderWindows` keeps only the first Finder window in focus order (most recently focused), stored as `lastFinderWindowId`. Focusing uses `launchOrFocusByBundleID`, which brings whatever window Finder considers frontmost. This is deliberately conservative.
